# UDA Task Corpus

This directory ships **benchmark tasks pre-migrated to the unified
`uda-desktop` container image**. Each immediate subdirectory holds one
benchmark; each subdirectory below it holds one task.

The corpus is generated by migration tools in the UDA-Gym repo
(`UDA-Gym/tools/migrate/<bench>.py`). The output lives here in
NanoRollout so the runtime path has no dependency on UDA-Gym at task
load time.

## Layout

```
nanorollout/envs/uda_env/adapter/
├── README.md                           # this file
├── __init__.py                         # exposes ADAPTER_ROOT
├── cocoa-v1/                           # 153 tasks from cocoabench-v1.0
│   ├── _migration_meta.json
│   ├── <task-id>/
│   │   ├── Dockerfile                  # FROM ghcr.io/bowenbryanwang/uda-desktop:<tag>
│   │   ├── docker-compose.yaml         # ${TASK_DOCKER_IMAGE_NAME} + ${HOST_PORT}:8080
│   │   ├── task.yaml(.enc)             # agent prompt
│   │   ├── test.py(.enc)               # verifier (test(result) -> dict)
│   │   ├── canary.txt                  # decryption salt
│   │   └── assets/                     # optional fixture files
│   └── ...
├── osworld-v2/                         # (TODO) the ~66% CLI-bypassable subset
└── swe-bench-mm/                       # (TODO) 612 frontend bug fixes
```

## Per-task schema

Every task subdirectory follows the same on-disk layout, regardless of
the source benchmark. This is intentional so that one rollout runner
(`nro run --task uda --agent <bench>`) can consume them all uniformly.

| File | Required | Purpose |
|---|---|---|
| `Dockerfile` | yes | Thin overlay on `uda-desktop`. Base image pinned by the migrator. |
| `docker-compose.yaml` | yes | Builds + runs the Dockerfile, maps `${HOST_PORT}:8080`. |
| `task.yaml` or `task.yaml.enc` | yes | Agent instruction (top-level key: `instruction`). |
| `test.py` or `test.py.enc` | yes | Verifier exporting `test(result: dict) -> {"passed": bool, "feedback": str, "details": dict}`. |
| `canary.txt` | iff encrypted | Per-task salt for the decryption helper. |
| `assets/` | no | Static fixture files copied at build time. |

## What the migrator changes vs. preserves

For `cocoa-v1`:

- **Rewrites** `FROM ghcr.io/agent-infra/sandbox:<tag>` → `FROM ghcr.io/bowenbryanwang/uda-desktop:<pinned-tag>`.
- **Rewrites** `/home/gem/` → `/home/kasm-user/` everywhere in the Dockerfile. The `uda-desktop` image also ships a `/home/gem -> /home/kasm-user` symlink as a safety net for paths hidden inside encrypted task assets.
- **Preserves byte-for-byte** all `docker-compose.yaml`, `task.yaml(.enc)`, `test.py(.enc)`, `canary.txt`, and `assets/`.
- **Writes** `_migration_meta.json` with the source revision and migrator version so the output is reproducible.

## Regenerating

From UDA-Gym workdir, with the upstream cocoa-agent checkout at
`cocoa-agent/`:

```bash
python tools/migrate/cocoa.py --src cocoa-agent/cocoabench-v1.0
```

The default `--dst` resolves to this directory's `cocoa-v1/`. Pass
`--dst` explicitly to write elsewhere (e.g. for a v2 alongside v1).
